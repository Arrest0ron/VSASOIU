# ISA Manual учебного процессорного ядра ASOIU-16

## 1. Назначение

ASOIU-16 - простое специализированное 16-битное процессорное ядро для учебного проекта по RTL-проектированию, верификации и синтезу. Ядро реализуется по Гарвардской архитектуре: память команд и память данных разделены.

Цели архитектуры:

- простой модульный RTL на Verilog;
- явное разделение Data Path и Control Unit;
- управляющий автомат в виде Moore FSM;
- достаточный набор команд для демонстрации арифметики, логики, памяти и ветвления;
- удобная автоматическая верификация через тестбенч.

## 2. Основные параметры

| Параметр | Значение |
| --- | --- |
| Разрядность данных | 16 бит |
| Разрядность команды | 16 бит |
| Количество регистров общего назначения | 8 |
| Разрядность регистра | 16 бит |
| Адресация памяти команд | по словам, 16 бит на команду |
| Адресация памяти данных | по словам, 16 бит на слово |
| Размер памяти команд в базовой реализации | 256 слов |
| Размер памяти данных в базовой реализации | 256 слов |
| Архитектура | Гарвардская |
| Тип Control Unit | Moore FSM |

## 3. Регистровый файл

Регистровый файл содержит 8 регистров `R0`...`R7`.

| Регистр | Назначение |
| --- | --- |
| `R0` | константа `0`; запись в `R0` игнорируется |
| `R1`...`R7` | регистры общего назначения |

Два порта чтения используются для операндов `rs1` и `rs2`. Один порт записи используется для результата `rd`. Запись выполняется по фронту тактового сигнала при активном `reg_we`.

## 4. Форматы команд

Все команды имеют длину 16 бит. Поле `opcode` всегда занимает старшие 4 бита `[15:12]`.

### 4.1 R-type

Используется для команд с двумя регистровыми операндами.

```text
15          12 11       9 8        6 5        3 2        0
+-------------+----------+----------+----------+----------+
| opcode[3:0] | rd[2:0]  | rs1[2:0] | rs2[2:0] | 000      |
+-------------+----------+----------+----------+----------+
```

Семантика: `rd <- f(rs1, rs2)`.

### 4.2 U-type

Используется для унарной логической команды `NOT`.

```text
15          12 11       9 8        6 5        0
+-------------+----------+----------+----------+
| opcode[3:0] | rd[2:0]  | rs1[2:0] | 000000   |
+-------------+----------+----------+----------+
```

Семантика: `rd <- ~rs1`.

### 4.3 I-type

Используется для загрузки константы и адресации памяти с базовым регистром.

```text
15          12 11       9 8        6 5                   0
+-------------+----------+----------+---------------------+
| opcode[3:0] | rd/rs2   | rs1/base | imm6                |
+-------------+----------+----------+---------------------+
```

Для `LOAD`: `rd <- MEM[rs1 + sign_extend(imm6)]`.

Для `STORE`: `MEM[rs1 + sign_extend(imm6)] <- rs2`, где поле `[11:9]` трактуется как `rs2`.

Для `ADDI`: `rd <- rs1 + sign_extend(imm6)`.

### 4.4 B-type

Используется для условного перехода `BEQ`.

```text
15          12 11       9 8        6 5                   0
+-------------+----------+----------+---------------------+
| opcode[3:0] | rs1[2:0] | rs2[2:0] | offset6             |
+-------------+----------+----------+---------------------+
```

Семантика: если `rs1 == rs2`, то `PC <- PC + sign_extend(offset6)`, иначе `PC <- PC + 1`.

Смещение считается относительно текущего `PC`. Для перехода на следующую команду используется `offset6 = 1`.

### 4.5 J-type

Используется для безусловного перехода.

```text
15          12 11                                      0
+-------------+-----------------------------------------+
| opcode[3:0] | addr12                                  |
+-------------+-----------------------------------------+
```

Семантика: `PC <- addr12[7:0]` в базовой реализации с памятью команд на 256 слов.

## 5. Система команд

| Мнемоника | Opcode | Формат | Описание |
| --- | --- | --- | --- |
| `NOP` | `0x0` | R-type | Нет операции; `PC <- PC + 1` |
| `ADD rd, rs1, rs2` | `0x1` | R-type | `rd <- rs1 + rs2` |
| `SUB rd, rs1, rs2` | `0x2` | R-type | `rd <- rs1 - rs2` |
| `AND rd, rs1, rs2` | `0x3` | R-type | `rd <- rs1 & rs2` |
| `OR rd, rs1, rs2` | `0x4` | R-type | `rd <- rs1 | rs2` |
| `XOR rd, rs1, rs2` | `0x5` | R-type | `rd <- rs1 ^ rs2` |
| `NOT rd, rs1` | `0x6` | U-type | `rd <- ~rs1` |
| `LOAD rd, [base + imm6]` | `0x7` | I-type | Загрузка слова из памяти данных |
| `STORE rs2, [base + imm6]` | `0x8` | I-type | Запись слова в память данных |
| `JUMP addr12` | `0x9` | J-type | Безусловный переход |
| `BEQ rs1, rs2, offset6` | `0xA` | B-type | Переход, если регистры равны |
| `ADDI rd, rs1, imm6` | `0xB` | I-type | `rd <- rs1 + sign_extend(imm6)` |
| `HALT` | `0xF` | R-type | Остановка процессора |

Команда `ADDI` добавлена как вспомогательная команда для удобной загрузки малых констант и написания тестовых программ. Обязательные команды из задания представлены полностью.

## 7. Флаги и условия

В базовой версии отдельный регистр флагов не используется. Условный переход `BEQ` сравнивает два регистра напрямую в Data Path.

Для отладки и тестирования Top Module предоставляет сигналы:

- `halted` - процессор остановлен командой `HALT`;
- `pc` - текущий счетчик команд;
- `instr` - текущая команда;
- `state` - текущее состояние управляющего автомата.

## 8. Модель выполнения

Ядро является многотактным. Одна команда проходит несколько состояний FSM:

1. `FETCH` - чтение команды из памяти команд по адресу `PC`.
2. `DECODE` - декодирование `opcode` и полей команды.
3. `EXECUTE` - выполнение операции ALU, вычисление адреса или проверка условия ветвления.
4. `MEMORY` - доступ к памяти данных для `LOAD`/`STORE`.
5. `WRITEBACK` - запись результата в регистровый файл.
6. `HALT` - остановка после команды `HALT`.

## 9. Пример программы

Пример проверочной программы вычисляет сумму `5 + 7`, сохраняет результат в память, загружает его обратно, проверяет `BEQ` и выполняет `HALT`.

```asm
ADDI  R1, R0, 5       ; R1 = 5
ADDI  R2, R0, 7       ; R2 = 7
ADD   R3, R1, R2      ; R3 = 12
STORE R3, [R0 + 10]   ; MEM[10] = 12
LOAD  R4, [R0 + 10]   ; R4 = 12
BEQ   R3, R4, 2       ; перейти через следующую команду
ADDI  R5, R0, 1       ; ошибка, если выполнится
HALT
```

Ожидаемый результат после завершения:

- `R3 = 12`;
- `R4 = 12`;
- `R5 = 0`;
- `MEM[10] = 12`;
- `halted = 1`.

## 10. Артефакты следующих этапов

На основе этой ISA на следующих этапах создаются:

- RTL-модули `alu`, `register_file`, `control_unit`, `processor_core`;
- тестбенч с автоматической загрузкой программы и проверкой результатов;
- временные диаграммы VCD/GTKWave;
- структурная схема архитектуры и диаграмма FSM;
- отчеты синтеза и симуляции.
